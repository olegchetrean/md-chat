Security policy

Version 1.0 · Effective 18 May 2026 · Next review: November 2026

This page describes how MD-Chat is engineered and operated for security, how to report a vulnerability responsibly, and how we comply with the EU Cyber Resilience Act (Regulation (EU) 2024/2847), which begins applying to vulnerability handling on 11 September 2026.

1. Reporting a vulnerability

If you believe you have found a security vulnerability in MD-Chat — server (Synapse fork), web client, mobile clients (Element X forks), AI layer, infrastructure, or documentation — please report it confidentially.

  • Email: security@md-chat.eu
  • PGP key: md-chat.eu/.well-known/pgp-key.txt (fingerprint published in SECURITY.md)
  • Encrypted Matrix DM: @security:md-chat.eu
  • security.txt: md-chat.eu/.well-known/security.txt (RFC 9116)

What to include

  • A clear description of the issue and its impact
  • Steps to reproduce, ideally with a minimal proof-of-concept
  • Affected versions, commits, or endpoints
  • Your contact information and a name for credit (optional)

Our commitments

  • Acknowledgement: within 24 hours, business or otherwise
  • Triage decision: within 72 hours
  • Status updates: every 7 days until resolution
  • Coordinated disclosure: standard 90-day window, negotiable for complex issues
  • Public credit: in release notes, advisories (CVE), and the Hall of Fame — with your consent
  • Safe harbour: we will not pursue legal action against good-faith researchers who comply with this policy (see section 6)

3. Operational security

  • Infrastructure hosted on Hetzner Falkenstein (Germany, EU) with full-disk encryption
  • SSH access by hardware token only (FIDO2 / YubiKey), no passwords, no shared accounts
  • All admin actions logged to an append-only audit log with hash-chained integrity
  • Production secrets in a dedicated secret store (sops + age, soon Vault); never in repository
  • Deploys via signed commits and signed container images; Sigstore cosign verification on production
  • SBOMs published per release (CycloneDX format), aligned with CRA Article 13
  • Dependency monitoring via Dependabot, GitHub Advanced Security, OSV-Scanner

4. Audits and testing

  • Annual independent penetration test (planned: Cure53 or Radically Open Security in Q1 2027)
  • Continuous fuzz testing on parser surfaces (ClusterFuzzLite)
  • Static analysis (Semgrep, CodeQL) on every PR
  • Dynamic analysis on staging before each minor release
  • Cryptographic implementations reviewed by community via public code

5. Bug bounty program

A public bug bounty program will launch alongside the v0.9 beta (Sprint 11, target Q4 2026). Until then, we offer:

  • Public credit and a swag pack for any in-scope finding
  • Discretionary monetary rewards for severe vulnerabilities, subject to budget
  • Priority triage for findings affecting cryptography, authentication, or data confidentiality

6. Safe harbour

We will not pursue civil or criminal action against security researchers who:

  1. Make a good-faith effort to avoid privacy violations, destruction of data, or service degradation
  2. Stop as soon as the vulnerability is confirmed and report it promptly
  3. Do not access, modify, or exfiltrate data belonging to other users beyond what is strictly necessary to demonstrate impact
  4. Give us reasonable time to remediate before public disclosure (default 90 days)
  5. Do not use the vulnerability for personal gain or to harm others

If you are uncertain whether a planned action is permitted, write to security@md-chat.eu first.

7. Incident response

If we confirm a security incident affecting users:

  • Internal triage within 1 hour, severity assigned (SEV-1 to SEV-4)
  • User notification within 72 hours where GDPR art. 33 is triggered; sooner for SEV-1
  • Authority notification: lead supervisory authority via the EU Representative; ENISA via the single CSIRT under CRA art. 14 (from 11 Sept 2026); CNPDCP Moldova if MD residents affected
  • Post-mortem published within 30 days for SEV-1 and SEV-2, redacted only as needed

8. Cyber Resilience Act alignment

From 11 September 2026, the EU Cyber Resilience Act requires manufacturers of products with digital elements to:

  • Apply a vulnerability handling process with disclosure obligations
  • Notify actively exploited vulnerabilities and severe incidents to ENISA within 24h (early warning), 72h (notification), 14 days (final report)
  • Provide SBOMs and security updates for the support period
  • CE marking for products in conformity

Our internal CRA readiness checklist tracks compliance against each obligation; we publish progress in the compliance docs.

9. Transparency

  • Semi-annual Transparency Report (legal requests, accounts affected, content removed): first edition January 2027
  • Security advisories: published as GitHub Security Advisories and assigned CVEs where applicable
  • Canary statement (warrant canary): updated monthly at md-chat.eu/canary
  • Architecture documents: docs/architecture.md
