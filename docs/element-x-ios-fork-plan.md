# Element X iOS Fork Plan — `md-chat-ios`

> Target audience: an iOS developer who has never shipped a Matrix client.
> Source: Element X iOS (`element-hq/element-x-ios`), branch `develop`,
> calendar-versioned (current upstream `26.05.x`).
> Output: `md-chat-ios` — sovereign EU messenger client for the MD-Chat platform.
> Default homeserver: `https://msg.md-chat.eu`.
> Compliance: EU AI Act Article 50, GDPR (RM Law 195/2024 from 23 Aug 2026), eEvidence.

---

## 1. Codebase overview

Element X iOS is a **SwiftUI** application that talks to a **Matrix homeserver**
through the **matrix-rust-sdk** via a generated Swift FFI. It is *not* the legacy
Element iOS (which was UIKit + matrix-ios-sdk written in Swift/ObjC). Everything
that touches the protocol — sync, encryption (Megolm + experimental MLS),
sliding sync, push decryption, room list, timeline — happens in Rust. Swift is
mostly UI, navigation, and platform glue.

### 1.1 Project generation: XcodeGen, not Tuist

`ElementX.xcodeproj` is **generated from `project.yml`** using
[XcodeGen](https://github.com/yonaskolb/XcodeGen). You almost never edit the
`.xcodeproj` by hand — the project file is technically committed (so CI can
build without a generation step) but every meaningful change goes through
`project.yml` plus the YAML fragments in `Tools/XcodeGen/`.

Bootstrap is one command:

```bash
swift run tools setup-project
```

This installs `xcodegen`, `swiftgen`, `swiftlint`, `swiftformat`, `sourcery`,
`mockingbird` and git hooks via Mint/Homebrew, then runs XcodeGen and the
`Tools/XcodeGen/postGenCommand.sh` hook (which patches the project with a
SwiftLint build phase, code coverage exclusions, and a Periphery aggregate
target).

### 1.2 Build system layers

| Layer | Source | Output |
|---|---|---|
| App targets | `project.yml` + Swift sources under `ElementX/` | `ElementX.app`, `NSE.appex`, `ShareExtension.appex` |
| Tooling | `Package.swift` at repo root | `swift run tools <subcommand>` CLI for setup, SDK build, string download, screenshots |
| Rust SDK | external SPM dep `matrix-rust-components-swift` (pre-built `MatrixSDKFFI.xcframework`) | linked into all three app targets |
| Localizations | `ElementX/Resources/Localizations/*.lproj/Localizable.strings` + `Untranslated.strings` synced via Localazy | SwiftGen generates `Strings.swift` from these |
| Assets | `ElementX/Resources/Assets.xcassets` + `Resources/AppIcon.icon` | SwiftGen generates `Asset.swift` enum |
| Design system | local SPM `compound-ios/` (vendored, not a git submodule) | `CompoundDesignTokens` + `Compound` Swift module |

### 1.3 Rust SDK linkage (critical to understand before forking)

Element X depends on `matrix-rust-components-swift` at an *exact* pinned
calendar version (e.g. `26.05.13`). That repo is a thin Swift package whose
`Package.swift` declares a `MatrixRustSDK` library backed by a remote
`MatrixSDKFFI.xcframework` zip hosted on Element's CDN. The xcframework
contains pre-compiled Rust for `ios-arm64`, `ios-arm64-simulator`,
`ios-arm64_x86_64-simulator`, and `macos-arm64_x86_64`.

For local SDK work you run:

```bash
swift run tools build-sdk
```

which clones `matrix-rust-sdk` into a sibling directory, builds the FFI for
all slices using `cargo` + `cross`, and rewrites the Swift package reference
to a local path. **You must understand this dual-mode (remote pinned vs local
path) flow before touching anything Rust.**

### 1.4 Target inventory

* **ElementX** — the iOS app (also runs on iPad and Mac Catalyst is disabled).
* **NSE** (`NotificationServiceExtension`) — decrypts encrypted push payloads
  before iOS displays them. Required for E2EE notifications. Has its own
  bundle ID `BASE_BUNDLE_IDENTIFIER.nse` and shares a keychain access group.
* **ShareExtension** — appears in the iOS share sheet; bundle ID
  `BASE_BUNDLE_IDENTIFIER.shareextension`.
* **UnitTests / UITests / IntegrationTests / PreviewTests / AccessibilityTests**.
* **SDKMocks** — mock layer over `MatrixRustSDK` for previews and tests.

### 1.5 Deployment target

iOS 18.5 (currently). That is *aggressive* — App Store rejection for low-end
device exclusion is not a concern in 2026, but be aware: anything below iOS 17
will not run the app at all.

## 2. Brand assets to replace

Element brand colours are dominated by the green **Compound Green**
(`#0DBD8B` / `#0BC491`) plus the white-on-green hourglass-logo `EX` glyph and
the marketing name "Element X". Every visual touchpoint must be replaced.

### 2.1 App icon & alternate icons

* `ElementX/Resources/AppIcon.icon` — iOS 18 *AppIcon Composer* format
  (layered, supports tinted/dark/clear variants). Replace the layered PSDs with
  MD-Chat artwork. Export sizes auto-generated by Xcode 16+.
* `ElementX/Resources/Assets.xcassets/Application/AppIcon-Dev.appiconset`,
  `AppIcon-Nightly.appiconset` — *alternate* icons used by debug, nightly and
  PR-preview builds. Replace each with MD-Chat-branded equivalents (different
  background tint per channel is the convention).

### 2.2 Splash / launch screen

Element X uses a **storyboard-less** launch via `UILaunchScreen` keys in
`Info.plist` (`UILaunchScreens` → `UILaunchScreenDefinition`) pointing to an
image set named `launch-logo` inside `Assets.xcassets`. Replace `launch-logo`
imageset (1x/2x/3x PDF or PNG) with MD-Chat wordmark. Background color is
defined in the `UIColorName` key — change to MD-Chat brand colour token.

### 2.3 Accent colours and design tokens

The accent colour is *not* defined in `Assets.xcassets` directly. It lives in
the **Compound design system** under `compound-ios/Sources/Compound/Tokens/`
which is generated from `compound-design-tokens` JSON. The single source of
truth file is `compound-ios/Sources/Compound/Tokens/CompoundColors.swift`.

Two paths:

1. **Quick rebrand (recommended initially)**: override only the semantic
   `accent` token via a custom `CompoundColors` extension in
   `ElementX/Sources/Other/Extensions/CompoundColors+MDChat.swift`. Element's
   colour use is mostly semantic (`compound.textActionAccent`,
   `compound.iconAccentTertiary`, `compound.bgAccentRest`), so a single
   override propagates everywhere.
2. **Full rebrand**: fork `compound-design-tokens` upstream, regenerate the
   Swift package, vendor `md-chat-design-tokens` into `compound-ios/`. Heavier
   but cleanest long-term.

Recommended palette for MD-Chat:

|---|---|---|
| `accentPrimary` | `#1F4FA8` (Moldova blue) | `#0DBD8B` (Element green) |
| `accentSecondary` | `#FFD24D` (warm yellow accent) | — |

## 3. String files to rebrand

### 3.1 Where strings live

* `ElementX/Resources/Localizations/<lang>.lproj/Localizable.strings`
* `ElementX/Resources/Localizations/<lang>.lproj/Localizable.stringsdict`
* `ElementX/Resources/Localizations/<lang>.lproj/InfoPlist.strings`
* `ElementX/Resources/Localizations/Untranslated.strings` —
  **the only file you edit by hand**. New English strings go here.

Element X has *not* migrated to `Localizable.xcstrings` (the catalog format)
yet — they still use legacy `.strings` files plus Localazy for sync. Plan
accordingly. (Apple's `xcstrings` migration is on the upstream roadmap; rebase
strategy in §14 must account for it.)

### 3.2 Code generation

`Tools/XcodeGen/postGenCommand.sh` runs `swiftgen` against
`swiftgen-config.yml` to produce:

* `ElementX/Sources/Generated/Strings.swift` — `enum L10n` with one static
  property per key, including pluralisations.
* `ElementX/Sources/Generated/Asset.swift` — `enum Asset` with image/color
  refs.

You never commit these (they are gitignored) — they regenerate on each XcodeGen
run and on every clean build via a "Generate Strings" build script phase that
calls `Tools/Scripts/generateStrings.sh`.

### 3.3 String key audit for rebrand

Search the entire `.strings` corpus for:

* `Element` (case-sensitive whole-word) — appears in ~120 keys. Replace with
  `MD-Chat`.
* `element.io` / `element.hq` URLs — ~40 occurrences. Replace with
  `md-chat.eu`.
* `matrix.org` — keep where it refers to the *protocol*; replace where it
  refers to the *default homeserver brand*.
* `Element Pro`, `EMS`, `Element Web` — remove entirely (these reference
  upstream commercial offerings).
* `Sign in to Element` / `Welcome to Element X` onboarding copy — full rewrite
  for the MD-Chat narrative.

Use `rg -i 'element' ElementX/Resources/Localizations/en.lproj/` to enumerate;
the resulting diff is the rebrand string PR.

### 3.4 Localazy considerations

Element runs `swift run tools download-strings` against their Localazy
project. **Do not use their Localazy project for the fork** — that would
pollute Element's translation memory. Either:

1. Create a separate Localazy project for `md-chat-ios` (free for OSS).
2. Or use Apple's String Catalog (`Localizable.xcstrings`) and skip Localazy
   entirely — Xcode 16 handles plurals and varies, and we ship in only 5–6
   languages.

**Recommendation: migrate to `Localizable.xcstrings` during the fork.** This
removes a vendor dependency, makes the rebase easier (no Localazy round-trip),
and aligns with Apple's direction.

## 6. Push notification migration

### 6.1 How Element X push works

iOS push to encrypted Matrix clients goes through **Sygnal** (the Matrix
push gateway). Flow:

1. App registers with APNs, receives device token.
2. App posts a *pusher* to the homeserver with `kind: "http"`,
   `data.url: "https://push.element.io/_matrix/push/v1/notify"`,
   `data.format: "event_id_only"`, `data.default_payload.aps.mutable-content: 1`,
   and `pushkey: <device-token>`.
3. When a message arrives, homeserver hits Sygnal, which sends a *silent*
   APNs payload to Apple.
4. APNs delivers to the device, iOS wakes the **NSE**, NSE decrypts the
   Matrix event via Rust SDK using the keychain-stored E2EE keys, NSE mutates
   the notification content with the decrypted body, system displays it.

### 6.2 What to change

* **Run our own Sygnal** at `https://push.md-chat.eu` (already on roadmap;
  Docker image `matrixdotorg/sygnal`). Provision APNs Auth Key `.p8`,
  configure Sygnal with `key_id`, `team_id` (MEGA PROMOTING's), `bundle_id`
  `eu.md-chat.messenger`, `app_id` `eu.md-chat.messenger.ios`.
* In `AppSettings.swift` change the pusher URL constant from
  `https://push.element.io/_matrix/push/v1/notify` to
  `https://push.md-chat.eu/_matrix/push/v1/notify`. Look for `pushGatewayURL`
  or `pusherProfileTag` properties.
* In `NSE/NotificationServiceExtension.swift` no change — decryption uses
  Rust SDK and shared keychain, gateway-agnostic.
* In `Info.plist` keep `aps-environment` = `production` (and `development`
  for the Debug scheme).

### 6.3 Voice/video call push (Element Call)

Element X uses **PushKit/VoIP** for incoming call invites if Element Call is
enabled. For MD-Chat we **disable Element Call in v1** (we ship Jitsi-free
audio/video later) — set `elementCallEnabled = false` in AppSettings and
remove the `com.apple.developer.usernotifications.communication` entitlement
and PushKit declaration. This avoids App Review questions about VoIP push
misuse.

## 7. Multi-language strategy

### 7.1 Element X language coverage

Element X currently ships 30+ locales via Localazy, including `en`, `fr`,
`de`, `it`, `es`, `nl`, `pt-BR`, `pl`, `cs`, `sk`, `uk`, `ru`, `tr`,
`zh-Hans`, `zh-Hant`, `ja`, `ko`, etc. Romanian is **partial** (~30 % coverage

### 7.2 What we ship in v1

* `en-GB` — base development locale.
* `ro` (Romanian) — **first-class, 100 % coverage required**.
* `ru` (Russian) — first-class, 100 % coverage required.
* `en-US` — auto-falls-back to `en-GB`, included for store listing.
* `uk` (Ukrainian) — second-class, partial coverage, opt-in. Useful for
  diaspora.

Everything else: remove from `Resources/Localizations/` to shrink IPA and to
avoid shipping stale Element-branded translations we cannot audit.

### 7.3 Migration to xcstrings catalog

Action plan:

1. Delete all `*.lproj/Localizable.strings` except `en.lproj/`, `ro.lproj/`,
   `ru.lproj/`, `uk.lproj/`.
2. Create `ElementX/Resources/Localizable.xcstrings` (Xcode → New File →
   String Catalog) and import the four sources via
   `xcrun extractLocStrings`.
3. Translate all ~3500 keys in Xcode's catalog editor. For `ro` budget ~40 h
   professional review (we have native speakers); for `ru` ~30 h; for `uk`
   crowd-source via community.
4. Remove Localazy from `project.yml` build phases; delete `localazy.json`.
5. Update `Tools/Scripts/generateStrings.sh` to no-op or to use SwiftGen's
   `xcstrings` parser (SwiftGen ≥ 6.6).

### 7.4 Region-specific strings (`ro-MD` vs `ro`)

Apple supports region modifiers (`ro-MD`, `ro-RO`). We ship **base `ro`** and
add a `ro-MD.lproj` override **only** for strings that differ in Moldovan
register (e.g. "Acasă" vs "Acasă", currency "MDL" vs "RON"). Maybe 20 keys
total. Saves duplicating 3500 strings.

Similarly `ru-MD` is not a recognised Apple region; we use plain `ru` and
override Moldova-specific strings in code via `Locale.current.region`.

## 8. Configuration defaults — homeserver and providers

### 8.1 Element X defaults

In `AppSettings.swift`:

```swift
let accountProviders = ["matrix.org"]
let allowOtherAccountProviders = true
```

The "account provider" is the homeserver shown on the sign-in screen as a
suggestion. `allowOtherAccountProviders = true` lets the user type any
homeserver URL.

### 8.2 MD-Chat defaults

```swift
let accountProviders = ["md-chat.eu"]
let allowOtherAccountProviders = false   // hard-locked in v1
let defaultHomeserverAddress = "https://msg.md-chat.eu"
```

We **lock** `allowOtherAccountProviders = false` for the App Store build
because:

* It enforces our sovereignty narrative (no accidental federation onboarding
  to matrix.org).
* It simplifies App Review (no "users can sign into arbitrary servers" Q).
* Federation across servers still works at the *protocol* level — users just
  can't *sign up* on a foreign server through our client.

For an enterprise/community variant we can flip the flag via a build
configuration (`MD_CHAT_OPEN_FEDERATION` xcconfig flag).

4. Publish the `MatrixSDKFFI.xcframework` zip to a GitHub release in
   `md-chat-rust-components-swift`.
5. In `md-chat-ios/project.yml`, change `MatrixRustSDK` URL to the fork.

**Important caveat:** the FFI surface changes between minor versions. MLS
adds methods on `Encryption`, `Room`, `Timeline`. We have to wrap these
behind protocols in the Swift codebase (`EncryptionProtocol`, `RoomProxy`) so
rebases don't shatter the app.

## 14. Release cadence and upstream rebase strategy

### 14.1 Upstream cadence

Element X iOS releases roughly **every 2 weeks** following calendar
versioning. The Matrix Rust SDK underneath releases a new tagged
xcframework every 1–2 weeks. Breaking FFI changes happen ~once per quarter.

### 14.2 Our cadence

* **Patch releases (`26.06.x`)**: as needed for bug fixes / compliance
  hotfixes. Out-of-band.
* **Minor rebases**: every **4 weeks** — pull from upstream `develop`,
  resolve conflicts, run full test suite, ship TestFlight beta.
* **Major rebases**: every **quarter** when FFI breaks. Allocate 3–5 days.

### 14.3 Rebase strategy

Maintain a **long-lived branch**: `md-chat/main`.

```
upstream develop ──────●────●────●────●────●────●──→
                        \         \         \
md-chat/main         ●───●───●─────●───●─────●───●──→
                     │   │   │     │   │     │   │
                  brand  cmp brand merge cmp merge cmp
```

Where `brand` = pure cosmetic commit, `cmp` = compliance feature commit,
`merge` = upstream sync. Keep brand and compliance commits **isolated and
revertable** so an upstream merge that conflicts can be repaired by reverting
our commit, merging, then re-applying.

Tooling: use `git rerere` (`git config rerere.enabled true`) so identical
conflicts auto-resolve over multiple rebases.

### 14.4 What stays in our fork forever

* `ElementX/Sources/Compliance/*` (new module, no upstream conflict).
* `eu.md-chat.*` bundle identifiers + entitlements.
* Brand assets + replaced strings.
* `Resources/Localizations/Localizable.xcstrings` (if we migrate away from
  Localazy).

### 14.5 What we contribute upstream

* Bug fixes encountered during integration → upstream PR to
  `element-hq/element-x-ios`.
* Generic xcstrings migration → upstream PR (already on their roadmap, our
  work helps).
* MLS / PQXDH wrapper protocols → upstream PR (good for ecosystem).

Contributing upstream **reduces our rebase debt**. It's a 5 % time
investment that saves 20 % rebase pain.

**(b) Top 5 risks:**

1. **App Store rejection — federation/UGC concern.** Apple has rejected
   Matrix clients before (FluffyChat, Cinny attempts) over moderation. Mitigate
   with a "report" flow, content blocking, 17+ rating, App Review notes
   explaining the moderation toolkit.
2. **Matrix Rust SDK FFI breakage.** MLS and PQXDH are experimental — every
   minor SDK release can break our Swift wrappers. Mitigation: wrap FFI
   behind protocols, run integration tests on every SDK bump, pin exactly.
3. **Apple Developer Program enrollment delay for Moldovan SRL.** D-U-N-S
   verification for Moldova takes 1–4 weeks. Start day 1.
4. **Localazy vs xcstrings drift.** Migrating off Localazy is the right move
   but creates a one-time merge conflict with every upstream rebase until
   Element themselves migrate. Could lose 4–8 h per rebase for 6 months.
5. **AI Act Art 50 misinterpretation.** Wording of the disclosure banner must
   be lawyer-reviewed; a wrong banner could create regulatory exposure worse
   than no banner. Engage Moldovan + EU counsel.

**(c) Recommended fork strategy:** Hard fork at upstream tag `26.05.13`,
maintain a single long-lived `md-chat/main` branch, rebase from upstream
`develop` every 4 weeks, isolate brand/compliance commits so they can be
reverted-and-re-applied during conflict resolution, contribute generic
improvements (xcstrings migration, MLS protocol wrappers) back upstream to
minimise long-term rebase debt. Do **not** attempt to keep the fork purely as
a runtime configuration overlay — Element does not expose enough configuration
for that, and the compliance UI requires real Swift code.
