## 1. Upstream codebase architecture

Element X Android is the second-generation Matrix client from Element (the Riot/Element rewrite). It is **not** Kotlin Multiplatform — it is an Android-native app written in **Kotlin 2.3.21** with **Jetpack Compose** UI, sharing its protocol layer with Element X iOS through the **matrix-rust-sdk** FFI bindings (`org.matrix.rustcomponents:sdk-android`). The Compose-multiplatform brief in the task description was slightly off: there is no `commonMain`; everything lives in `:app`, `:features:*`, `:libraries:*`, `:services:*`, `:appconfig`, `:appicon`, `:appnav`, `:plugins`, `:codegen`.

### Build system

- **Gradle 8.x with Kotlin DSL** (`build.gradle.kts` everywhere).
- **AGP 8.13.2**, **Kotlin 2.3.21**, **KSP 2.3.7**.
- **JVM target Java 21** (toolchain), **compileSdk 36**, **targetSdk 36**, **minSdk 24** (FOSS) / 33 (enterprise variant — we ignore that flavor).
- **Compose BOM 2026.05.00**, navigation by **Appyx** (Bumble's tree-based Compose nav library, not Jetpack Nav).
- Custom Gradle plugins in `:plugins/` enforce architecture rules (`konsist` tests), version bumping (CalVer: `(2000+year)*10000 + month*100 + releaseNum` → e.g. `26260101`), Localazy sync, dependency analysis.
- Static analysis: **detekt + ktlint + Android lint + konsist**. CI fails on any.

### Module taxonomy

```
:app                              ← single Application module, wires everything
:appconfig                        ← compile-time config: homeservers, learn-more URLs, feature flags
:appicon                          ← launcher icon resources only
:appnav                           ← top-level NavTarget definitions for Appyx
:features:login / :logout / :verifysession / :signedout / :linknewdevice / :deactivation
:features:home / :ftue / :migration / :announcement
:features:messages / :startchat / :invite / :invitepeople / :forward / :share
:features:createroom / :joinroom / :leaveroom / :roomdetails / :roomdetailsedit
:features:roomdirectory / :roomaliasresolver / :roomcall / :roommembermoderation
:features:reportroom / :space / :knockrequests
:features:call                    ← Element Call (WebRTC, MatrixRTC) embedded WebView
:features:userprofile / :preferences / :lockscreen / :securebackup / :securityandprivacy / :rolesandpermissions
:features:poll / :location / :viewfolder
:features:analytics / :rageshake / :networkmonitor / :cachecleaner / :licenses
:features:enterprise              ← stub for paid variant; we strip
:libraries:matrix:api + :impl     ← thin wrapper around matrix-rust-sdk
:libraries:pushproviders:api + :firebase + :unifiedpush
:libraries:designsystem           ← Compound design tokens (Element brand)
:libraries:ui-strings             ← shared string resources
:libraries:cryptography / :sessionstorage / :preferences / :network / ...
:services:appnavstate / :toolbox / :analytics / :analyticsproviders:*
:tests:konsist / :uitests
```

Each feature module has the `:api` / `:impl` / `:test` triplet; `:api` exposes only Dagger entry-point interfaces, `:impl` is loaded via Anvil. **Dependency injection: Anvil + Dagger 2** (not Hilt — Element X switched away).

### matrix-rust-sdk dependency

- Pinned to **`org.matrix.rustcomponents:sdk-android:26.05.13`** in `gradle/libs.versions.toml`. This bundles a prebuilt `.aar` containing native `.so` for `arm64-v8a`, `armeabi-v7a`, `x86_64`, `x86`, plus the Kotlin bindings generated by UniFFI.
- The Rust SDK is itself the **matrix-rust-sdk** repo (`matrix-org/matrix-rust-sdk`) — single source of truth for sync, E2EE, key backup, sliding sync. It is where MLS and PQXDH will land (see §10).

## 3. String resource rebrand & localization

Element strings live in **two layers**:

1. `libraries/ui-strings/src/main/res/values*/` — shared strings, **machine-managed by Localazy**. Editing `strings.xml` directly is allowed only for temporary additions; canonical edits go through `tools/localazy/` scripts.
2. Per-feature module `src/main/res/values*/strings.xml` — local strings for that feature.

### Rebrand task

We must replace **all literal occurrences** of:
- `"Element"`, `"Element X"`, `"element.io"`, `"matrix.org"` (default homeserver — see §8), `"@element"` social handles.
- The strings `app_name`, `app_name_short`, `legal_*`, `branding_*`, `welcome_*_title`, `welcome_*_subtitle`, `learn_more_url_*`.
- The `error_failed_loading_element` family (yes, "Element" is hard-coded in error copy).

**Approach:** a single Gradle task `replaceBrandStrings` in `:plugins` that scans `*/src/main/res/values*/strings.xml` and applies a YAML allow-list of substitutions (`brand-substitutions.yaml`). Run it once at fork-time; subsequent upstream merges replay the task. Translatable strings get **`translatable="false"`** for the brand-name itself (`app_name`, `app_name_short`) so Localazy does not localize "MD-Chat".

### Language variants to add

Element ships ~40 languages via Localazy, including `ro` and `ru`. Both `values-ro/` and `values-ru/` **already exist** in `libraries/ui-strings`. What we add:

- **`values-ro-rMD/`** (Moldovan Romanian — currently inherits from `ro`). New strings for MD-Chat-specific copy (eEvidence disclosures, AI Act §50 banner, KYC for VLOP-style operator account, GDPR export). The `-rMD` qualifier lets us override only the brand-sensitive strings while inheriting the bulk from `ro`.
- **`values-ru/`** for Transnistria + diaspora — already shipped, we add MD-Chat-specific strings only.
- **`values-uk/`** — already in upstream, kept as-is. Strategic: Ukrainian diaspora is the second-largest MD-Chat target.
- **`values-en-rGB/`** — UK English for EU institutional users (regulator outreach). Only formality tweaks ("organisation" vs "organization").

We **do not** localize to Gagauz or Bulgarian at v1.0 — deferred to v1.1 once user base validates demand. Documented in `docs/roadmap.md`.

### Localazy strategy

Upstream Element owns the `element` Localazy project; we do **not** push to it. Instead, we run a **private Localazy project** `md-chat-android` that imports only our delta strings (MD-Chat-specific keys). At build time, `tools/localazy/sync.py` (forked & rewritten) pulls Element's public translations from their Localazy CDN export endpoint and merges with our delta. This avoids paying Localazy for translation of strings we did not author.

## 6. F-Droid alternative track

F-Droid is non-negotiable: it is the trust signal for the privacy-conscious users we target (journalists, activists, diaspora). The challenge: F-Droid bans Google proprietary blobs, which means **no Firebase Cloud Messaging**.

### Strategy: UnifiedPush

[UnifiedPush](https://unifiedpush.org/) is a standardized push protocol with **distributor apps** (NextPush, ntfy, Conversations) and gateway support in Matrix Synapse via [`matrix-sygnal`](https://github.com/matrix-org/sygnal) UnifiedPush mode. Element X already has a UnifiedPush implementation in `:libraries:pushproviders:unifiedpush` — we just enable it in the F-Droid flavor.

### F-Droid build requirements

- **Reproducible build**: F-Droid requires byte-for-byte reproducibility. We must publish the exact `Versions.kt`, fix the keystore-less APK signature, and commit a `metadata/eu.mdchat.android.yml` to [`fdroiddata`](https://gitlab.com/fdroid/fdroiddata).
- **No proprietary deps**: the `fdroid` flavor must exclude Firebase, PostHog (proprietary SDK — replace with opt-in self-hosted PostHog), Sentry (kept — Sentry SDK is MIT and points to our self-hosted Sentry instance at `sentry.md-chat.eu`), Google Play Services Location (replace with `MicroG` fallback — Element X already handles this).
- **`Anti-Features` flag**: F-Droid will flag us with `NonFreeNet` (because we federate over Matrix which uses public servers) and possibly `Tracking` (PostHog even opt-in is debated). We pre-negotiate with F-Droid reviewers.
- **F-Droid metadata YAML**: `Categories: Internet`, `License: AGPL-3.0-or-later`, `WebSite: https://md-chat.eu`, `SourceCode: https://github.com/olegchetrean/md-chat-android`, `IssueTracker: …`, `Changelog: …/blob/main/CHANGELOG.md`, `AutoUpdateMode: Version`, `UpdateCheckMode: Tags`.
- **Self-hosted F-Droid repo (recommended)**: in addition to mainline F-Droid (slow review cycle, ~2-6 weeks), we publish to our own F-Droid repo at `https://fdroid.md-chat.eu/repo` using [`fdroidserver`](https://gitlab.com/fdroid/fdroidserver) on the CI runner. Users add it via QR code in 10 seconds. This gives us same-day updates while waiting for mainline.

## 7. Multi-flavor strategy

Element X's existing `store` flavor dimension is exactly what we need. We rename and tune:

```kotlin
android {
    flavorDimensions += "distribution"
    productFlavors {
        create("gplay") {
            dimension = "distribution"
            applicationIdSuffix = ""           // production = eu.mdchat.android
            buildConfigField("String", "FLAVOR_DESC", "\"Google Play\"")
            buildConfigField("boolean", "ENABLE_FIREBASE", "true")
            buildConfigField("boolean", "ENABLE_UNIFIEDPUSH", "true")  // user choice
        }
        create("fdroid") {
            dimension = "distribution"
            applicationIdSuffix = ".fdroid"     // eu.mdchat.android.fdroid — coexists on device
            buildConfigField("String", "FLAVOR_DESC", "\"F-Droid\"")
            buildConfigField("boolean", "ENABLE_FIREBASE", "false")
            buildConfigField("boolean", "ENABLE_UNIFIEDPUSH", "true")
        }
    }
}

dependencies {
    "gplayImplementation"(libs.firebase.messaging)
    "gplayImplementation"(libs.firebase.bom)
    implementation(libs.unifiedpush.connector)  // both flavors
}
```

Source set discrimination:
- `app/src/gplay/AndroidManifest.xml` declares `<service android:name=".push.fcm.FirebaseMessagingService">`.
- `app/src/fdroid/AndroidManifest.xml` omits FCM service entirely.
- `app/src/main/kotlin/.../push/PushProviderRegistry.kt` reads `BuildConfig.ENABLE_FIREBASE` and registers conditionally.

CI builds **both** flavors on every PR. Release publishes `gplay` AAB to Play Console and `fdroid` APK to our self-hosted F-Droid repo + a GitHub Release.

## 10. matrix-rust-sdk pinning for MLS + PQXDH

**Current upstream pin:** `sdk-android:26.05.13`.

**Forward path:**
- **MLS (RFC 9420)** in Matrix is tracked in [MSC4143](https://github.com/matrix-org/matrix-spec-proposals/pull/4143) and the [`matrix-rust-sdk` mls branch](https://github.com/matrix-org/matrix-rust-sdk/tree/feat/mls). As of May 2026 it is **experimental** — single-room MLS works, federation across MLS-disabled homeservers does not. We watch for the `mls/v0.1` tagged release.
- **PQXDH (CRYSTALS-Kyber hybrid)** — Signal's quantum-safe key agreement, being integrated into vodozemac (Matrix's E2EE crate). Tracking issue: [vodozemac#160](https://github.com/matrix-org/vodozemac/issues/160). Expected stable Q3 2026.

**Our pinning strategy:**
1. Pin to **`sdk-android:26.05.13`** at v1.0 (same as upstream).
2. Build CI matrix runs against **`sdk-android:main-SNAPSHOT`** nightly to detect breaking changes early.
3. Maintain a **`md-chat-sdk-android`** patched build (Rust source fork at `olegchetrean/matrix-rust-sdk`) only if we need to ship MLS before upstream tags a release. Acceptance criterion: at least one independent security review on the MLS implementation before enabling it for any user.
4. Native libs: matrix-rust-sdk ships `.so` for `arm64-v8a`, `armeabi-v7a`, `x86_64`, `x86`. We keep all four — `x86_64` covers Chrome OS, `armeabi-v7a` covers cheap entry phones used by Moldovan rural diaspora.
5. APK split-by-ABI: yes, both flavors, reduces download size by ~40 MB per ABI.

## 11. Compliance UI

The reason this fork exists is regulatory differentiation. Three compliance surfaces:

### AI Act §50 (Article 50 — transparency obligations for AI systems)

- Triggered when a user enters a room flagged `io.mdchat.ai_assisted = true` (set via custom room state event by room admins who deploy MD-Chat AI assistants — the Kallina / aichat.md tie-in).
- Modal on first message, dismissible-once-per-room: "This room contains an AI-assisted participant. Messages may be processed by an AI service operated by MEGA PROMOTING S.R.L. under [AI Act §50 disclosure](https://md-chat.eu/legal/ai-act-50)."
- Persistent banner above the composer for AI-flagged rooms.
- New string keys: `compliance_ai_act_50_modal_title`, `..._body`, `..._learn_more`, `compliance_ai_act_50_banner`.
- New Compose component: `:libraries:designsystem` → `AiDisclosureBanner.kt`.

### eEvidence (EU Regulation 2023/1543 — cross-border production orders)

- A "Legal hold export" flow in `:features:preferences` → `Privacy & security` → `Legal & data`. Generates a signed PDF + ZIP of a user's own messages in a specified room/time-range, with a SHA-256 manifest, suitable for submission to authorities **by the user** (we do not respond to eEvidence orders directly — see `docs/compliance.md` §3).
- Link to `https://md-chat.eu/legal/eevidence` explaining MEGA PROMOTING's procedural posture.

### GDPR data export & deletion

- **Export**: invokes `POST /_matrix/client/v3/account/export` on the homeserver (Synapse module we ship server-side). Returns a tar.gz of all user events + media. App saves via Storage Access Framework.
- **Delete**: invokes `POST /_matrix/client/v3/account/deactivate` with `erase: true`. Server-side, the Synapse `erase_user` admin task purges room state where possible (subject to Matrix's append-only ledger constraints — documented to user in the deletion modal).
- New screens: `GdprExportScreen`, `GdprDeleteAccountScreen`, both in `:features:preferences:impl`.

## 12. ProGuard / R8 implications

Element X uses **R8 in full mode** (`android.enableR8.fullMode=true`) with aggressive optimization. Forking introduces three risks:

1. **Anvil-generated Dagger components** are reflection-light, but the matrix-rust-sdk UniFFI bindings use **JNI native names** that must be kept:
   ```pro
   -keep class uniffi.** { *; }
   -keep class org.matrix.rustcomponents.sdk.** { *; }
   -keepclassmembers class * { native <methods>; }
   ```
   Upstream already includes these in `app/proguard-rules.pro`. We must not strip them when consolidating rules.

2. **Sentry / PostHog SDK** rules. PostHog 3.43 introduced new `kotlinx.serialization` types. We add:
   ```pro
   -keep class com.posthog.** { *; }
   -keepattributes Signature, *Annotation*, EnclosingMethod
   ```

3. **UnifiedPush distributor lookup** uses reflection on installed packages — generally not affected by R8 (system API) but the connector library's `BroadcastReceiver` subclasses must not be obfuscated:
   ```pro
   -keep class org.unifiedpush.android.connector.** { *; }
   -keep class * extends org.unifiedpush.android.connector.MessagingReceiver
   ```

4. **Compose Compiler metrics**: enable `-Xverbose-phases` once in CI to confirm stable composition keys after rename. Failed rename = recomposition storms.

We add `:plugins:proguardRulesAggregator` Gradle task that concatenates per-module `consumer-rules.pro` files and lints for duplicates.

R8 baseline profile: we **generate a baseline profile** via the Macrobenchmark library (`:tests:benchmark`) on cold-start, room-list scroll, message-send. Ships in `app/src/main/baseline-prof.txt`. Reduces cold-start by ~25 % on Pixel 4a, more on entry-level devices common in MD diaspora.

## 14. Wear OS / Android Auto

**Decision: defer both to post-v1.2.**

- **Wear OS**: Element X has no Wear OS module. Building one means: a Compose-for-Wear UI, a `:wear` module, a dedicated companion app design (notify-only or full chat). Effort: ~6 engineer-weeks. ROI on MD-Chat target audience: very low. Documented in `docs/roadmap.md` as v1.3 candidate, contingent on EU institutional users requesting it.
- **Android Auto**: messaging apps are eligible for Android Auto via the [`MessagingService`](https://developer.android.com/training/cars/messaging) API. Element X does not implement it. Effort: ~2 engineer-weeks for read-aloud + reply-by-voice. ROI: moderate (commuter use case for the EU institutional cohort). Deferred to v1.2; requires "Cars" category Play approval which adds 2-4 weeks of review.

In v1.0 manifest we **explicitly declare** `<meta-data android:name="com.google.android.gms.car.notification.SmallIcon" />` is absent and we set `<uses-feature android:name="android.hardware.type.automotive" android:required="false" />` to keep the app installable on automotive head-units even without Auto support — they get the regular phone UI.

## 15. Release cadence vs upstream

Element X follows **CalVer monthly** (`YY.MM.N`). They tag a release roughly every 4 weeks, with point releases for security fixes.

**MD-Chat cadence:**

- **`upstream-merge` branch** auto-PRs every Monday from upstream `develop`. Reviewed and merged Wednesday if CI green.
- **Stable releases** every 6 weeks (slower than upstream — gives us a 2-week soak window after each upstream tag). Tag format `vYY.MM.N-mdchat.M`, e.g. `v26.05.1-mdchat.1`.
- **Security releases** within 72 h of an upstream security advisory affecting our code path. Bypass the 6-week cadence.
- **Public changelog** at `https://md-chat.eu/changelog/android` with separate "Upstream changes" and "MD-Chat changes" sections.
- **Feature freeze before EU institutional procurement windows** (Sept–Oct, March–April) — only security/bug fixes during these windows.

**Merge conflict hotspots** (where we expect 80 % of upstream merge pain):
- `appconfig/AuthenticationConfig.kt` — homeserver constants.
- `app/build.gradle.kts` — flavor & dependency edits.
- `libraries/designsystem/.../theme/*` — Compound token overrides.
- `app/src/main/AndroidManifest.xml` — intent filter renames.
- String files in `libraries/ui-strings` — brand replacements.

For each hotspot we ship a `.gitattributes` `merge=ours` declaration on the brand-substitution YAML file, and a CI job that re-runs `replaceBrandStrings` after every merge so the deltas are mechanical.

## Appendix A — File-by-file fork checklist

A reproducible 1-day checklist for the engineer doing the fork:

1. `gh repo fork element-hq/element-x-android olegchetrean/md-chat-android --clone`.
2. `git checkout -b mdchat-init`.
3. Run `scripts/init-fork.sh` (we author):
   - Replace `applicationId` and `:app` namespace.
   - Drop `:features:enterprise`, remove from `settings.gradle.kts`.
   - Rename `appconfig` constants.
   - Generate brand-substitution YAML.
   - Replace mipmaps from `brand/android/` directory.
4. `./gradlew :app:assembleGplayDebug :app:assembleFdroidDebug` — both must pass.
5. Konsist tests will fail on namespace mismatches — add exemptions in `tests/konsist`.
6. Install both APKs on a single device — verify they coexist (different `applicationId`s).
7. Smoke test login against `msg.md-chat.eu` (server team must have it up first).
8. Commit, push, open PR `mdchat: initial fork`.
